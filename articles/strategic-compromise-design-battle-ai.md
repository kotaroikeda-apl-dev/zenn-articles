「インターフェースは小さくあるべき」

この手の規約を初めて学んだとき、自分もかなりしっくりきていました。
責務は細かく分ける。依存は薄く保つ。抽象に依存する。きれいです。説明もしやすい。

でも、実務でコードを読むほど、少しずつ違和感が出てきました。

規約どおりに書かれていないコードがある。しかも、それを書いたのは雑な人ではなく、むしろ設計を一番分かっているリードエンジニアだったりする。

最初は「なんで崩したんだろう」と思いました。
ただ、読み進めて、AIにも構造を整理させて、最後に本人に聞いてみると、見えてくる景色が変わりました。

この記事で書きたいのは、規約を破るのは悪ではない、という雑な話ではありません。

**規約は北極星だけど、現場のコードはいつも最短距離では進めない。**
そのズレをどう読むか。そこに設計の実務がある、という話です。

## 1. イントロ: 美しい規約と、泥臭いコード

Go でもアーキテクチャの話でも、よく出てくる理想があります。

- インターフェースは小さくあるべき
- 1インターフェース1責務であるべき
- 実装の都合を上位レイヤーに漏らすべきではない

例えば、こういう形はとても美しいです。

```go
type UserReader interface {
    FindByID(ctx context.Context, id string) (*User, error)
}

type UserWriter interface {
    Save(ctx context.Context, user *User) error
}
```

必要な契約だけ切り出していて、読む側も迷いにくい。
「これで十分では？」と思います。

ところが現場では、こういうコードに出会います。

```go
type ExternalServiceBundle struct {
    UserAPI        UserAPI
    BillingAPI     BillingAPI
    NotificationAPI NotificationAPI
    AuditAPI       AuditAPI
    FeatureFlags   FeatureFlagClient
}

type CreateOrderUseCase struct {
    repo     OrderRepository
    services ExternalServiceBundle
}
```

きれいな小さいインターフェースではなく、巨大な Service バンドル。
規約だけ見れば、かなり気持ち悪いです。

自分も最初に見たときは、「いや、これはまとめすぎでは？」と思いました。
でも同時に、そこで止まると設計の理解は浅いままです。

ここで出てくる問いが、この記事の出発点です。

**なぜ、規約を作った人がそれを破るのか？**

まとめ: きれいな規約に反するコードこそ、現場の意思決定を読む入口になる。

## 2. レイヤーごとの「規律」のグラデーション

この違和感を整理するうえで大事だったのが、**全レイヤーで同じ厳しさのルールを適用しない**という見方でした。

特に分かりやすかったのは、ドメイン層とユースケース層の温度差です。

### ドメイン層は厳格でいい

ドメイン層では、細かい契約が効きます。

- ルールの意味が明確
- 影響範囲を狭く保ちたい
- 長く残る知識を濁らせたくない

例えば、こういう小さいインターフェースは強いです。

```go
type Clock interface {
    Now() time.Time
}

type OrderNumberGenerator interface {
    Next() string
}
```

このレイヤーで巨大バンドルを使うと、何に依存しているのかが一気にぼやけます。
なので、ドメイン層は「1インターフェース1メソッド」くらいの厳しさでもわりと戦えます。

### ユースケース層は柔軟でいい

一方で、ユースケース層は少し事情が違います。

ユースケース層がやっているのは、純粋なビジネスルールの定義だけではなく、複数の依存先を並べて処理を進めることです。
つまり、**オーケストレーション**です。

```go
type ServiceBundle struct {
    UserAPI         UserAPI
    InventoryAPI    InventoryAPI
    PaymentAPI      PaymentAPI
    ShipmentAPI     ShipmentAPI
    NotificationAPI NotificationAPI
}

type PurchaseUseCase struct {
    repo     OrderRepository
    services ServiceBundle
}
```

この形は、美しさだけで見ると負けています。
でも、ユースケース層が「複数サービスの流れを束ねる場所」だと割り切ると、見え方が変わります。

- 依存先を1か所に集められる
- 呼び出し順の文脈を追いやすい
- 配線の量を減らせる

もちろん何でも束ねていいわけではありません。
ただ、ユースケース層に求める役割がオーケストレーションなら、ドメイン層と同じ規律をそのまま持ち込むと、逆に不自然になることがあります。

レイヤーごとに守るべきものが違う。だから、ルールの厳しさもグラデーションになる。
ここが腹落ちすると、現場のコードの見え方がかなり変わります。

まとめ: ドメイン層は純度を守り、ユースケース層はオーケストレーションのしやすさを優先することがある。

## 3. 引数爆発（Argument Explosion）

Service バンドルが雑に見えても、現場で採用される理由の1つがこれです。

**引数爆発。**

依存関係を全部きれいに分けて注入すると、理論上は正しい。でも、実装の肌感としてはしんどくなることがあります。

例えばこんなコンストラクタです。

```go
func NewCreateOrderUseCase(
    orderRepo OrderRepository,
    userRepo UserRepository,
    txManager TxManager,
    userAPI UserAPI,
    inventoryAPI InventoryAPI,
    paymentAPI PaymentAPI,
    shipmentAPI ShipmentAPI,
    couponAPI CouponAPI,
    pointAPI PointAPI,
    notificationAPI NotificationAPI,
    auditAPI AuditAPI,
    logger Logger,
    tracer Tracer,
    metrics Metrics,
    featureFlags FeatureFlagClient,
    clock Clock,
    idgen IDGenerator,
    lockManager LockManager,
    retryPolicy RetryPolicy,
    config Config,
) *CreateOrderUseCase {
    // ...
}
```

20個近い依存が並ぶと、もうそれだけで読む気力が削られます。

しかもつらいのは、コードの美しさだけではありません。DX が壊れます。

- コンストラクタ呼び出しで毎回配線がつらい
- テストのセットアップが重い
- 依存追加のたびに引数の並びを壊しやすい
- IDE 補完だけでは全体像を掴みにくい

そこで、こういうまとめ方が出てきます。

```go
type OrderUseCaseServices struct {
    UserAPI         UserAPI
    InventoryAPI    InventoryAPI
    PaymentAPI      PaymentAPI
    ShipmentAPI     ShipmentAPI
    NotificationAPI NotificationAPI
    AuditAPI        AuditAPI
    FeatureFlags    FeatureFlagClient
}

func NewCreateOrderUseCase(
    orderRepo OrderRepository,
    txManager TxManager,
    services OrderUseCaseServices,
    logger Logger,
    clock Clock,
) *CreateOrderUseCase {
    // ...
}
```

これは依存を隠している、とも言えます。
実際、その批判は正しいです。

ここで出てくるのが、**依存関係の隠蔽 vs 可視性** のトレードオフです。

可視性を最大化すると、何に依存しているかは明確になります。
でも、構築コストや読解コストが跳ねる。

隠蔽を進めると、呼び出しは楽になります。
でも、内部で何を握っているのかがぼやける。

重要なのは、「どちらが正義か」ではなく、「今どの痛みを減らしたいか」です。

もしそのユースケースが外部APIを何本も束ねる層なら、全部を1個ずつ広げるより、バンドル化した方が開発体験としてマシな場面は普通にあります。

まとめ: 引数爆発を避けるためのバンドル化は、怠慢ではなく DX を守るための設計になることがある。

## 4. リードエンジニアへの確認

ここまで考えても、まだ自分の中では結論が出ませんでした。

なので素直に聞きました。

「この設計、理想からは少し外れて見えるけど、なぜこうしたんですか？」

返ってきた答えは、かなり実務的でした。

### 1. その時期はスピード優先フェーズだった

当時はプロダクトの立ち上げ直後で、まずは機能を出して検証を回すフェーズだったそうです。

- 完璧な境界分離より、まず出す
- リファクタ前提で進める
- 将来の一般化より、今の変更速度を優先する

この話を聞いて、設計を静的な正解として見ていた自分の視点がズレていたと気づきました。

設計は、いまこの瞬間のプロダクト戦略と無関係ではないんですよね。

### 2. 外部 API ラッパーという制約があった

もう1つ大きかったのが、ユースケースの多くが「外部 API を束ねるラッパー」に近い性格だったことです。

つまり、そのユースケースはドメインルールを濃く表現する場所というより、

- API A を呼ぶ
- 結果を整形する
- API B に渡す
- 失敗時に通知する

といった、接続と順序制御の比重が大きい場所だった。

その場合、依存を小さく分割しすぎても、本質的な複雑さは消えません。
むしろ、オーケストレーションを行う層なのに、きれいに見せるためだけに配線コストを増やすことになります。

ここでようやく腑に落ちました。

**設計は時間軸で変わる。**

立ち上げ期、拡大期、安定運用期では、最適な設計判断は同じではない。
ある時点で正しかった戦略的妥協が、半年後には負債になることもあるし、その逆もあります。

だから、コードレビューで大事なのは「原則違反を見つけること」だけではなく、
**この違反は雑さか、戦略か** を見極めることなんだと思います。

まとめ: 設計の良し悪しは形だけでは決まらず、その時期の速度要求と制約を見ないと判断を誤る。

## 5. 結論: 規約は北極星、コードは航跡

ここまでの話を雑にまとめると、こうです。

- 規約は理想を示す
- コードは現場の意思決定の履歴を残す

だから、規約からズレたコードを見たときに「ダメな実装だ」と即断すると、学びを1段浅いところで止めてしまいます。

むしろ見るべきなのは、そのズレです。

- なぜズレたのか
- どの痛みを減らしたかったのか
- その判断は今も有効なのか

この問いを持てると、コードリーディングが一気に実務的になります。

そして、ここで AI はかなり役に立ちます。
ただし、AI に答えを丸投げする使い方では弱いです。

自分がいま有効だと感じているのは、次の学び方です。

1. AIで構造を理解する
2. ズレを見つける
3. 仮説を立てる
4. 人に聞く

例えば、

- 「この UseCase は何をオーケストレーションしているか」
- 「この ServiceBundle を分解すると何が困るか」
- 「依存のまとまりは責務単位として自然か」

みたいな問いは、まず AI に壁打ちできます。

でも最後は、人に聞く。
なぜなら、設計の背景にはたいてい、ロードマップ、組織事情、障害経験、期限、API制約みたいな、コードだけでは見えない文脈があるからです。

AI は構造を読むのがうまい。
人は意思決定の理由を知っている。

この2つをつなぐと、学習効率が一気に上がります。

規約を守るだけでは、強くなりません。
規約と現実のズレを読めるようになって、やっと現場の設計が分かってくる。

それが今の自分の実感です。

まとめ: 規約は進む方角を示し、コードはそこへ向かう途中で下した判断の跡を残している。

## 明日からやること

- 気になる「規約違反」を見つけたら、すぐ直す前に「何の痛みを減らしているのか」を1回書き出す
- AI に「この設計のズレは何か」「別案のコストは何か」を聞いて、自分の仮説を作ってからレビューや相談に行く
- リードや実装者に「なぜそうしたか」を聞くときは、正誤判定ではなく時間軸と制約を確認する

設計は、正しさの暗記ではなく意思決定の読解です。
その視点を持つだけで、現場のコードの見え方はかなり変わるはずです。
