## 早期最適化は、昔より今の方が危ない

> 数msのために、チーム全員の理解を失うコードは高すぎる。

いま自分が早期最適化を危険だと感じる理由は、まさにこれです。

「早期最適化は良くない」という話自体は、もうかなり有名です。
でも今の現場では、この話は昔よりずっと重い意味を持っています。

昔は CPU やメモリが本当に貴重でした。
少ないリソースでいかに動かすかは、そのままプロダクトの成立条件だった時代があります。

一方で今は、クラウドで計算資源をある程度買えます。
もちろん無限ではないし、性能設計が不要になったわけでもありません。
ただ、多くの現場で本当に足りなくなっているのは、CPU より**人間の理解力**です。

- 誰が読んでも意図が分かるか
- 3か月後の自分が安全に直せるか
- オンコール中にすぐ原因を追えるか
- チームの他メンバーがレビューできるか
- 新しく入ったメンバーがオンボーディングで追いつけるか

いま高いのは、計算資源よりもこのコストです。
だからこそ、早期最適化は「少しコードが読みにくくなる」程度の話では済みません。
**人間の認知資源を先に食い潰す設計判断**になりやすい。

AI によって実装コストは激減しました。
でも、理解コストは下がっていません。
むしろレビュー、保守、オンボーディングの価値は相対的に上がっています。

つまり今の現場では、書く速度より**理解速度**の方が重要です。

この記事では、単なる教科書的な「早期最適化は悪」という話ではなく、AI 時代の実務として、なぜそれが性能改善より理解負債を増やしやすいのかを整理します。

## 1. 早期最適化とは何か

### 早期最適化の定義

早期最適化とは、**本当のボトルネックがまだ分かっていない段階で、性能を気にして複雑な実装を入れてしまうこと**です。

例えばこんなものです。

- まだ遅いと判明していないのにキャッシュを入れる
- まだ重くないのにループをトリッキーに圧縮する
- 実測もないのにメモリアロケーション削減を優先する
- 将来の負荷を想像して、抽象化や非同期処理を過剰に入れる

要するに、問題が観測される前に、問題がある前提でコードを歪めることです。

### 「早期最適化はあらゆる悪の根源」

有名な言葉として、Donald Knuth の

> Premature optimization is the root of all evil.

があります。

この言葉は雑に消費されがちですが、本質は「最適化するな」ではありません。
**最適化すべき場所が分かる前に、全体を複雑化するな**という話です。

実務で危ないのは、最適化そのものより、最適化のタイミングを間違えることです。

### なぜ人は最適化したくなるのか

これはかなり自然な欲求です。

- 速いコードは賢く見える
- 余計な処理を見ると削りたくなる
- 「この実装、なんか無駄そう」が気になってしまう
- 将来スケールしたときの不安を先に潰したくなる
- ベンチマークの数字が改善すると達成感がある
- 技術的に一段上のことをやっている感覚がある

しかも、最適化は局所的には気持ちいいんですよね。
数行変えて速くなると、仕事した感が強い。

実務だと、ここにもうひとつ厄介な感情があります。

「後で遅いって言われたくないから、先に潰しておきたい」

この気持ちはすごく分かります。
でも、その善意がそのまま理解負債になることがある。
ここが早期最適化のいやらしいところです。

でも実務では、その数 ms の裏で

- 読みやすさ
- 修正しやすさ
- レビューしやすさ
- 障害対応しやすさ

を失っていることがある。

ここが落とし穴です。

## 2. なぜ危険なのか

### 可読性が落ちる

早期最適化されたコードは、たいてい素直に読めません。

例えば、まずは普通の実装です。

```go
func ActiveUsers(users []User) []User {
    result := make([]User, 0)
    for _, user := range users {
        if user.DeletedAt == nil && user.Enabled {
            result = append(result, user)
        }
    }
    return result
}
```

これは遅いかもしれません。
でも、ほとんどのエンジニアが 10 秒で理解できます。

一方で、最初から最適化欲が強く出ると、こうなりがちです。

```go
func ActiveUsers(users []User) []User {
    result := make([]User, 0, len(users))
    for i := 0; i < len(users); i++ {
        u := users[i]
        if (u.DeletedAt == nil) && u.Enabled {
            result = append(result, users[i])
        }
    }
    return result[:len(result):len(result)]
}
```

この程度ならまだ軽傷です。
でも現場ではさらに、

- 不要にポインタを使う
- 再利用用のバッファを共有する
- 分岐を潰して意図を読みにくくする
- 先に micro-optimization のイディオムを持ち込む

みたいなことが起きます。

するとレビュー時に「これ、何をしたかったんだっけ？」から始まる。

性能改善は局所的なのに、**読みにくさはコード全体の体験を悪化させる**のが厄介です。

### 実際にどんなコードが“理解負債”になるのか

「理解負債」というと抽象的に聞こえますが、現場ではだいたい次のような形で現れます。

- キャッシュの整合性ルールがコードから読めない
- 並列化した理由が計測なしで、今も必要か分からない
- メモリ最適化のための再利用オブジェクトが複数箇所で共有されている
- 例外ケースだけ妙に速くするための分岐が増えている
- コメントを読まないと意図が分からず、そのコメントも古い

こういうコードは、動いている間はまだいいんです。
問題は、仕様変更か障害対応が来た瞬間です。

「これ、どこまで触って大丈夫なんだっけ？」

という時間が長いコードは、だいたい理解負債です。

実務で価値が高いのは、ベンチマークの 1 回勝ちではなく、チーム全員が安全に変更できる時間の長さです。

### 保守性が下がる

保守では、書くときより直すときの方が圧倒的に長いです。

例えば最初からキャッシュを入れたコードは、性能上は良さそうに見えます。
一見すると「DB アクセスを減らして速くするだけ」です。
なので、最初の一手としてかなり魅力的に見えます。

でも実際には、ここで設計の性質が変わります。
なぜなら、**同じデータが DB とメモリ上の 2 箇所に存在する状態**になるからです。

この瞬間に、単なる高速化の話ではなく、整合性の話が始まります。

- どちらが正なのか
- いつ同期するのか
- 更新されたら何を消すのか
- 不整合が起きたらどう検知するのか

つまりキャッシュは、速さを得る代わりに、データの一貫性を自前で扱う責任を持ち込みます。

```go
type UserService struct {
    cache map[string]*User
}

func (s *UserService) Find(id string) *User {
    if user, ok := s.cache[id]; ok {
        return user
    }

    user := fetchUser(id)
    s.cache[id] = user
    return user
}
```

最初は単純でも、実務ではすぐに次の話が乗ってきます。

例えば、ユーザー情報が更新されたとします。
このとき DB の値だけ更新して、メモリ上のキャッシュを消し忘れると、アプリは古いユーザー情報を返し続けます。

ローカルでは気づきにくいのに、本番では「たまに古い名前が返る」「権限変更がすぐ反映されない」みたいな、嫌な不具合になります。

さらに複数サーバー構成になると、話はもっと面倒です。
あるサーバーではキャッシュが更新されたのに、別のサーバーには古いキャッシュが残ることがある。
すると、どのサーバーに当たるかで返るデータが変わる、というかなり厄介な状態になります。

- 更新時の invalidation はどうするのか
- 複数プロセスで整合性は取れるのか
- テストでキャッシュ状態をどう初期化するのか
- stale なデータで事故らないのか
- 障害時に切り戻しやすいのか

テストでも前提条件が増えます。
キャッシュが空のときの結果と、キャッシュが残っているときの結果で挙動が変わるので、毎回「このテストはどのキャッシュ状態を前提にしているのか」を意識しないといけない。

これが地味に効きます。
テストが不安定になったり、単体では通るのに並列実行で落ちたりすると、原因が仕様なのかキャッシュ状態なのか切り分けづらくなる。

障害対応でも同じです。
キャッシュを無効化してまず DB の値だけで振る舞いを確認する、という逃げ道が設計上用意されていないと、原因の切り分けがかなり難しくなります。

「DB が壊れているのか」「キャッシュが古いのか」「特定ノードだけおかしいのか」が分からないまま調査が長引く、というのは現場で普通にあります。

つまり、1回の「速そうな改善」が、ずっと面倒を見続ける仕組みに化ける。

言い換えると、

**キャッシュは単なる高速化ではなく、データ整合性の問題を持ち込む設計判断である**

ということです。

ここを軽く見て、「とりあえず速くなりそうだから」で入れると、後でかなり高くつきます。

### 修正時の心理的コストが増える

これは実務でかなり大きいです。

複雑な最適化コードって、触る側が怖いんですよね。

- なんでこの実装なのか分からない
- でも速さのためにこうしていそう
- 下手に直すと性能が落ちそう
- もっと悪いと、レアケースで壊れそう

結果として、「本当は直した方がいい違和感」が放置されます。
コードの問題が、技術的問題ではなく**心理的な近寄りにくさ**に変わる。

これはかなりまずいです。

チーム開発で強いのは、賢いコードではなく、**誰でも安全に触れるコード**です。

### バグ調査が難しくなる

最適化が早すぎるコードは、障害調査でもつらいです。

例えば、

- キャッシュがある
- 非同期キューがある
- オブジェクト再利用がある
- 条件分岐を潰した特殊処理がある

こういうコードで不具合が出ると、因果関係を追うのが一気に難しくなります。

ユーザー報告としては「たまに値が古い」「特定条件だけ二重送信される」程度なのに、内部では

- いつキャッシュに入ったのか
- どの goroutine が書き換えたのか
- このフラグは最適化の都合か仕様か

を全部見ないといけない。

しかも、こういう不具合は再現率が低い。
再現率が低い上に、読むコードは難しい。
最悪です。

### チーム開発で理解できる人が減る

これが一番大きい問題かもしれません。

早期最適化は、しばしば「書いた本人だけが文脈を持っているコード」を作ります。

レビューでもよくあるのが、

- 「たぶん速くするためなんだろうけど、意図までは追いきれない」
- 「動いてるなら触らない方がよさそう」
- 「詳しい人がいるときに見てもらおう」

という空気です。

つまり、コードがチームの資産ではなく、特定個人の専有知識になる。

その状態で担当者が異動したり退職したりすると、一気に負債化します。

数 ms 速いコードより、**全員が理解できるコードの方が価値が高いケースは本当に多い**です。
特に業務システムや SaaS の多くは、この現実を無視できません。

毎秒数百万リクエストをさばくホットパスでもない限り、まず問うべきは

> このコード、チーム全員が読めるか？

です。

## 3. AI時代でさらに危険になった理由

ここが、今の現場で一番重要だと思っています。

### AIで実装コストは下がった

いまは AI によって、コードを書くコストがかなり下がりました。

- 雛形はすぐ出る
- テストコードもそれなりに出る
- リファクタ案も複数出る
- ベンチマーク用コードもすぐ作れる

つまり、「実装すること」自体の希少性は下がっています。

昔なら半日かかった実装のたたき台が、今は数分で出る。
これは本当に大きい変化です。

でも、その反面で見落としやすいことがあります。

### 理解コストは下がっていない

コード生成は速くなっても、**そのコードを理解して責任を持つコスト**はあまり下がっていません。

むしろ下手をすると、上がっています。

AI はそれっぽい最適化を平気で提案します。

- 先回りしたキャッシュ
- まだ不要な並列化
- 過剰に汎化された抽象化
- 局所ベンチだけ見て入れた micro-optimization

これらは「書く」のは簡単です。
でも、「なぜ必要か」「今も必要か」「安全に直せるか」を判断するのは、人間の仕事です。

ここが重い。

つまり AI 時代は、実装コストが下がったせいで、**理解不能なコードを生産する速度まで上がった**とも言えます。

### 複雑なコードはAIでも扱いづらい

「AI があるなら複雑でもいいのでは？」という感覚は、現場だとかなり危険です。

AI は複雑なコードベースでも助けになります。
ただし、複雑さに強いわけではありません。

例えば、

- 副作用が複数箇所に散っている
- 最適化のための例外分岐が多い
- コメントがないと意図が読めない
- コメントはあるが、もう実装とズレている

こういう状態だと、AI も文脈の復元にコストがかかります。
結果として、

- もっともらしい誤読をする
- 一部だけ直して別の条件を壊す
- 安全側ではなく、派手な変更を提案する

ということが起きる。

つまり、複雑な最適化コードは**人間に厳しいだけでなく、AI にも厳しい**です。

AI を活かしたいなら、なおさらベースのコードは素直であるべきです。

### 人間がレビューするコストの方が高い

AI 時代のボトルネックは、書くことよりレビューです。

PR を出すのは速い。
でもレビューする側は、次のような問いに答えないといけません。

- 本当にこの最適化は必要か
- どの計測結果に基づいているか
- 可読性や保守性とのトレードオフは妥当か
- 障害時に追跡できるか

この判断は、生成よりずっと高いコストです。

最近の現場だと、コードが書けないことより、**レビューできる人間が足りないこと**の方がボトルネックになりやすい。

そうなると価値が高いのは、「AI がたくさんコードを書けること」ではなく、

**人間が短時間で正しさを判断できること**

です。

早期最適化は、ここに真っ向から逆行します。

### “読むコスト”がボトルネックになっている

今は本当にこれです。

書く速度は上がった。
実装候補も増えた。
でも、読む時間は増えていない。

1日でレビューできる PR の本数は急に 10 倍にはなりません。
オンコール中に読めるコード量も増えません。
新しく入ったメンバーがシステムを理解する速度も、魔法みたいには上がりません。

オンボーディングでも同じです。
AI がコードを量産できるほど、新しく入った人が読むべきコードの総量と複雑さも増えやすい。
このとき足を引っ張るのが、早期最適化された読みにくいコードです。

書くのが速いチームが強いのではありません。
**読むのが速いチームの方が、長期では強い**です。

だから今の現場では、

- 書くコストより読むコスト
- 実装コストより理解コスト
- マシンコストより人間コスト

が支配的です。

昔は CPU やメモリが貴重だった。
でも今の多くの開発現場では、**一番貴重なのは人間の理解力**です。

この前提に立つと、早期最適化は「ちょっと賢い工夫」ではなく、
**最も高い資源を先に消費する行為**に見えてきます。

### 今、価値が上がっている能力

今の時代に価値が上がっているのは、コードを書く能力だけではありません。

**他人が理解できる形で残す能力**

です。

これは地味ですが、かなり重要です。

- 誰が見ても意図が追える命名
- 素直なデータフロー
- 局所化された責務
- 限定された最適化
- 説明しなくても読める設計

こういうコードは、人間にも AI にも優しい。
結果として、開発速度が長期的に落ちません。

逆に、1人の気合いと文脈で成立している高速コードは、短距離では勝てても長距離で失速します。

AI 時代に評価されるのは、「速く書ける人」だけではなく、
**他人が理解できる形で速く前に進められる人**だと思っています。

## 4. ではどうすべきか

### まずは綺麗に作る

最初の実装では、まず素直に作るのが基本です。

- 意図が追える
- テストしやすい
- ログを入れやすい
- 後から分割しやすい

この状態を先に作る。

最初から速さを狙って構造を歪めると、あとで正しく直すのが難しくなります。
まずは綺麗に作って、普通に動かす。
ここを雑に飛ばさない方が、結局速いです。

### 計測してから最適化する

最適化の前に必要なのは、想像ではなく計測です。

- p95 / p99 が遅いのか
- DB が遅いのか
- ネットワークが支配的なのか
- アプリの CPU 使用率が高いのか
- GC が効いているのか

ここを見ずに最適化しても、当たりを外すことが多いです。

実務だと、アプリコードを頑張って速くしたのに、実際のボトルネックは外部 API や SQL だった、みたいなことは本当によくあります。

### ボトルネックが判明してから改善する

「遅いらしい」ではなく、「ここが遅い」と分かってから手を入れる。

これだけで、かなり事故が減ります。

例えばレスポンスが遅いなら、

1. まず計測する
2. どのレイヤーが支配的かを見る
3. 最小の変更で一番効くところを触る
4. 変更後にもう一度計測する

この流れを外さない。

勢いで複雑なキャッシュを入れる前に、インデックス 1 本で終わることもあるし、N+1 を潰すだけで十分なこともある。
本当に多いです。

### 最適化は局所的に行う

必要な最適化まで否定するつもりはありません。
ただし、やるなら**局所的に閉じ込める**べきです。

例えば、

- ホットパスだけ専用実装にする
- コメントで「なぜ必要か」を残す
- ベンチマークや計測結果を PR に添える
- 通常系のコードパスは素直なまま保つ

これが大事です。

コードベース全体を読みづらくするのではなく、必要な場所だけ難しくする。
そして、その難しさには根拠を持たせる。

### コメントよりシンプルな設計を優先する

「複雑だけどコメントを書けば読める」は、だいたい長持ちしません。

コメントは大事です。
でも、コメントがないと理解できない設計は、まず疑った方がいい。

理想は、

- コメントがなくてもだいたい読める
- コメントは背景や制約だけ補足する
- 実装の逐語訳はコメントにしない

です。

早期最適化されたコードは、しばしば大量の説明を必要とします。
でも本当に強いコードは、説明量が少ない。

設計がシンプルなら、コメントは補助で済みます。
設計が複雑だと、コメントは延命措置にしかならないことが多いです。

特に AI と一緒に開発するなら、これはかなり重要です。
AI はコメントも読みますが、最終的に扱いやすいのはコメント量の多いコードではなく、構造が素直なコードです。
複雑さをコメントで説明するより、複雑さそのものを減らした方が強い。

## 5. 実務でのリアル

ここまでかなり強めに書きましたが、もちろん理想論だけではありません。

### 本当に重い処理では最適化は必要

性能が事業に直結する領域では、最適化は普通に必要です。

- 金融の低レイテンシ処理
- 高トラフィックな広告配信
- 大規模検索
- 動画処理
- ゲームサーバー

このあたりでは、数 ms やメモリ効率が本当に効きます。

「読めればいい」はさすがに雑です。
そこは現実として切り分けるべきです。

### ただし、その場合でも計測が先

重要なのは、性能が大事な領域でも、やっぱり計測が先だということです。

本当に強いチームは、

- 負荷特性を知っている
- SLO / SLA を持っている
- 計測基盤がある
- どこで最適化すべきかを会話できる

状態になっています。

つまり、最適化を感覚でやらない。

性能が重要な現場ほど、むしろ

**どこを難しくしてよいかに厳しい**

です。

全部を速くしようとはしません。
効く場所だけを、根拠を持って削る。
それ以外は、ちゃんと読めるように保つ。

### 実務のあるある

現場でよくあるのは、こんな流れです。

- 将来怖いから先にキャッシュを入れる
- まだトラフィックは来ていない
- 数か月後、仕様変更で整合性問題が出る
- 誰も安全に直せず、結局キャッシュを剥がす

あるいは、

- 速そうだから並列化する
- 実は I/O 待ちが支配的で効いていない
- でも goroutine と同期処理だけは増える
- バグ調査だけ難しくなる

これ、珍しい話ではありません。

「最適化したのに得していない」は、実務だとかなり頻出です。

しかも最近は、AI がこの手の「それっぽい改善案」を高速で出せてしまいます。
だからこそ、提案をそのまま採用するのではなく、

- 本当に必要か
- 読みにくくならないか
- 3か月後に説明できるか
- 計測結果で正当化できるか

を人間が止まって見る必要があります。

## まとめ

早期最適化が危険なのは、単にコードが汚くなるからではありません。

今の開発現場では、

- AI によって実装コストが下がった
- でも理解コストは下がっていない
- むしろレビュー、保守、オンボーディングのコストが相対的に重くなった

という変化があります。

だから今は、昔以上に

**数 ms の性能改善より、チーム全員が理解できるコードの方が価値が高い**

ことが重要です。

昔は CPU やメモリが貴重だった。
でも今は、多くの現場でそれ以上に**人間の理解力が貴重**です。

速いけれど誰も触れないコードより、
そこそこ速くて、全員が理解できて、安全に改善できるコードの方が強い。

自分は、いまの時代の設計力はここに出ると思っています。

最初から賢く作ることではない。
**まず素直に作り、計測し、必要なところだけを賢くすること。**

これが、AI 時代の実務的な最適化だと思います。
