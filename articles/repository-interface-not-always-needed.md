「Repository は `interface` にするもの」

自分も最初は素直にそう信じていました。
クリーンアーキテクチャや DDD を学ぶと、Repository はインターフェースで定義して、実装を分けて、DI で注入する構成がきれいに見えます。実際、その設計がしっかりハマる場面はあります。

ただ、実務を重ねるほど、「毎回そこまでやるのが本当に最適なのか？」という疑問も出てきました。

特に小〜中規模の開発では、将来の拡張性のために入れた抽象化が、今の開発速度やコード理解のしやすさをむしろ下げることがあります。この記事では、クリーンアーキテクチャを否定するのではなく、**Repository のインターフェース化は常に正義ではない**という話を、現場感ベースで整理してみます。

## よくある構成

まずは、よくある Repository 構成です。Go でもかなり見かけます。

- `repository` にインターフェースを置く
- `repositoryimpl` あるいは `infra` に実装を書く
- Service にはインターフェースを注入する
- テストでは mock を差し込む

例えば、こんなイメージです。

```go
type UserRepository interface {
    FindByID(ctx context.Context, id string) (*User, error)
    Save(ctx context.Context, user *User) error
}
```

```go
type userRepository struct {
    db *sql.DB
}

func (r *userRepository) FindByID(ctx context.Context, id string) (*User, error) {
    // ...
    return nil, nil
}
```

```go
type UserService struct {
    repo UserRepository
}
```

この構成は見た目がきれいですし、依存関係も説明しやすいです。
ただ、現場で長く運用していくと、いくつかの違和感が出てきます。

## 違和感1: DB や ORM の差し替えって本当にそんなにある？

Repository をインターフェースにする理由として、よく「将来 DB クライアントや ORM を差し替えやすくするため」が挙げられます。

でも、自分の経験では、ここはかなり頻度が低いです。
`database/sql` から別 ORM に全面移行したり、MySQL から別の DB に載せ替えたりするのは、ゼロではないですが、日常的に起きる変更ではありません。

しかも、実際にその変更が起きるときは、Repository の実装だけを差し替えて終わることはあまりありません。

- クエリの書き方が変わる
- トランザクションの扱いが変わる
- ロード戦略が変わる
- エラーの種類やハンドリングが変わる
- パフォーマンス特性が変わる

つまり、インターフェースがあれば簡単に差し替えられる、というほど単純ではないことが多いです。
「いつかあるかもしれない差し替え」のために、最初から毎回抽象化するのは、少し先回りしすぎかもしれません。

## 違和感2: mock のためだけなら別手段もある

もう一つの定番理由が「テストしやすくするため」です。これも分かります。
インターフェースがあると mock を差し込みやすいので、ユニットテストは書きやすくなります。

ただ、ここも冷静に見る必要があります。

Repository を mock した Service テストばかり増えると、実際の SQL やクエリの妥当性は十分に担保されません。テストは全部通るのに、本番 DB につないだら壊れる、というのは普通に起きます。

それに Go では、最初から Repository 全体を抽象化しなくても、**使う側で必要最小限のインターフェースを切る**というやり方ができます。

これは Go でよく言われる "accept interfaces, return structs" に近い考え方でもあります。つまり、受け取る側だけが必要な契約を小さく定義し、返り値や実体は過度に抽象化しない、というやり方です。

本当にテストのために抽象化が必要なら、そのタイミングで小さく導入すればいい。最初から「Repository は必ずインターフェース」と決め打ちしなくても、テスト戦略は十分組めます。

## 違和感3: 実装にたどり着くまでが地味に遠い

これは日々のコードリーディングでかなり効いてきます。

インターフェース化された構成だと、処理を追うときにこうなりがちです。

1. Service からインターフェース定義へ飛ぶ
2. 実装を探す
3. DI 設定を見る
4. 実装ファイルを開く

設計としては分離されていても、読む側からするとちょっと遠いんですよね。

特に、少人数チームで素早く機能追加したいときや、新しく参加したメンバーがコードを追うとき、この「数ステップの遠さ」は思った以上に効きます。

最近は AI にコードを読ませたり、変更影響を一緒に考えたりする機会も増えました。だからこそ、**実装へ素直に辿り着ける構造**の価値は以前より上がっていると感じます。

抽象化は大事です。ただ、抽象化によって文脈が散ってしまうなら、読みやすさの面ではマイナスになることもあります。

## 違和感4: 小規模 PJ には重いことがある

小〜中規模のプロジェクトだと、インターフェース化によって増えるものは意外と多いです。

- インターフェース定義ファイル
- 実装ファイル
- DI の配線
- mock 生成の設定
- mock 自体の保守

一つひとつは小さくても、積み重なると確実に認知コストになります。

そして、実装が 1 つしかなく、今後も複数実装になる見込みが薄いなら、その抽象化は「将来のため」より「今の複雑化」の色が強くなります。

設計原則としてきれいでも、プロジェクト規模に対して重すぎる構成は、保守性を上げるどころか逆に複雑性を増やします。

## あえてインターフェース化しない構成

実務では、まず具象型で素直に始める方が合う場面もかなりあります。

例えば Go なら、こんな形です。

```go
type UserRepository struct {
    db *sql.DB
}

func NewUserRepository(db *sql.DB) *UserRepository {
    return &UserRepository{db: db}
}

func (r *UserRepository) FindByID(ctx context.Context, id string) (*User, error) {
    // ...
    return nil, nil
}
```

サービス層からはそのまま直接使います。

```go
type UserService struct {
    repo *UserRepository
}

func NewUserService(repo *UserRepository) *UserService {
    return &UserService{repo: repo}
}

func (s *UserService) GetUser(ctx context.Context, id string) (*User, error) {
    return s.repo.FindByID(ctx, id)
}
```

もちろん、Go で具象型をそのまま持つ場合は、パッケージ構成によっては循環参照が起きやすくなることがあります。なのでこの形は、同じパッケージ内に閉じる場合や、依存方向が素直な小〜中規模構成で特に取りやすい形です。

逆に、パッケージ境界を厳密に分けたい場合は、無理に具象型依存を押し通すより、どこで契約を切るべきかを設計として判断した方が自然です。

この構成の良いところは、単純に追いやすいことです。

- 実装がその場にある
- IDE で直接ジャンプしやすい
- ファイル数が少ない
- DI がシンプル
- 新メンバーにも説明しやすい

実装が 1 つしかない間は、単純であること自体がかなり強いです。

## それでもインターフェース化した方が良いケース

ここまで書くと「じゃあインターフェースは不要なのか」と見えそうですが、もちろんそうではありません。自分も次のようなケースではインターフェース化を選びます。

### 複数実装がある

最初から複数実装が見えているなら、インターフェースは自然です。

- 本番は DB、テストでは in-memory
- 顧客（tenant）ごとに保存先やアクセス方法が異なる
- 同じデータでも、DB・外部 API・検索エンジンなど複数の読み取り元がある

この場合は「将来あるかも」ではなく、すでに抽象化の必要があります。

### 外部 API や DB の切り替え予定が現実的にある

数か月以内に移行計画がある、段階的移行をする、というように変更が具体的なら、先にインターフェースを切る意味があります。

### 大規模組織で責務分離が重要

チームや担当境界が明確な組織では、インターフェースが契約として効きます。
依存方向の制御やレビュー責務の整理にメリットが出やすいです。

### テスト戦略上、本当に必要

高速にユースケーステストを回したい、外部依存をきっちり分離したい、という理由が強いならインターフェース化は有効です。

一方で、インターフェースを使わない場合でも、Repository 周りの検証手段がなくなるわけではありません。例えば `sqlmock` でクエリの挙動を確認したり、Testcontainers などで実 DB に近い形のテストを回したりするやり方もあります。

ただしこの場合も、「何となく全部抽象化」ではなく、必要性を明確にした方がうまくいきます。

## 結論

クリーンアーキテクチャや DDD の原則は大事です。自分も、依存方向を意識することや責務を分けること自体はとても重要だと思っています。

ただ、設計原則は毎回そのまま当てはめるものではなく、**現実の変更頻度・チーム規模・速度要求に合わせて選ぶもの**だと感じています。

実装が 1 つしかなく、差し替え予定も薄く、小〜中規模の開発でスピードが重要なら、Repository をあえてインターフェース化しない方が、むしろ読みやすく保守しやすいこともあります。

何でも抽象化すれば良いわけではありません。
特に AI 時代は、コードを「きれいに分けること」だけでなく、**人間も AI も理解しやすいこと**の価値が上がっています。

だからこそ、Repository のインターフェース化も「とりあえず入れる」ではなく、「本当に必要か」で決める。
そのくらい現実的な設計判断の方が、実務ではちょうどいいと思っています。
